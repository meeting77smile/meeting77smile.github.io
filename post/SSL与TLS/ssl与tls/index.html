SSL/TLS

1. 概念

  • SSL（Secure Socket Layer）全称安全套接字层协议，是Netscape公司率先采用的网络安全协议。它是在传输通信协议（TCP/IP）上实现的一种安全协议，采用公开密钥技术。SSL广泛支持各种类型的网络。

  • TLS（Transport Layer Security）全称传输层安全协议，前身为SSL协议。

SSL目前已经过时了，而当前的主流是TLS，但由于SSL过于出名，因此经常把TLS与SSL混着叫。

2. 加密算法

SSL和TLS的过程均同时使用了非对称和对称加密算法，因此有必要先了解这两种类型的算法

2.1 对称加密算法

加密算法是信息安全的核心技术，主要分为​​对称加密​​和​​非对称加密​​两大类，它们在密钥管理和应用场景上有着根本性的区别。

2.1.1 概念

  • 对称加密算法指在加密和解密过程中使用同一个密钥，因此也成为共享密钥加密。

  • 优点是计算量小、加密速度快和效率高，适合对大量数据进行加密 。

  • 缺点是难以分发和管理密钥（为了安全：要确保密钥不能明文发送；为了确保通信双方必须共享地使用同一个密钥：发送方和接收方的密钥是相同的）

2.1.2 常见的对称加密算法

  • AES (Advanced Encryption Standard) ：高级加密标准，支持128、192、256位三种密钥长度，是目前使用最广泛的对称加密标准，具有安全 、高效 的特点。

  • DES （Data Encryption Standard）：数据加密标准，一种早期的分组加密算法（即对数据按照相同的长度进行分组再加密）支持52位的密钥长度，容易被暴力破解，因此安全性较差，目前已经过时。

  • 3DES（Triple Data Encryption Standard）：是DES的增强版，即通过对每个数据块进行三次DES加密来提高安全性，虽然比DES安全，但处理速度较慢，目前也正被AES取代。

  • RC4 ：一种流密码算法，曾因速度快被广泛使用（如WEP协议），但后来发现了严重的安全漏洞，已不再安全，不应继续使用。

2.2 非对称加密算法

2.2.1 概念

非对称加密算法使用一对密钥：一个公钥（Public Key）和一个密钥（Private Key）。公钥可以公开给任何人（可以明文传输），私钥必须严格保密。公钥和私钥都可分别用于加密和解密，但对应的用途不同：

  • 用于加密数据：使用公钥进行加密，只能用对应的 私钥进行解密。

  • 用于数字签名 ：使用 私钥进行签名，可以用 公钥进行验证。

对称加密算法的优缺点分别是：

  • 优点 ：解决了密钥分发的问题（如接收方一开始同时具有公钥和私钥，公钥可以通过明文传输至发送方而私钥不需要发送，只需要保留在接收方出即可；之后发送方再通过公钥加密数据发给接收方，接收方便可使用私钥对数据进行解密。此过程不需要担心公钥泄露，因为即使得知了公钥，没有私钥也无法对数据进行解析）。

  • 缺点：计算复杂，加密速度远低于对称加密算法。

2.2.2 常见的非对称加密算法

  • RSA  (Rivest–Shamir–Adleman)​​：以三位提出者的首字母命名的非对称加密算法，是目前最经典、使用最广泛的非对称加密算法，其安全性基于大整数因数分解的数学难题。RSA既可用于加密/解密，也可用于数字签名，是SSL/SSH等安全协议的基础。

  • ECC (Elliptic Curve Cryptography)​​：椭圆曲线密码学，能够以较短的密钥长度实现与RSA同等的安全性，因此在计算资源受限的条件下（如移动设备、物联网）中更具优势​，效率更高。

  • DSA (Digital Signature Algorithm)：数字签名算法，只能用于签名，不能用于加密。

  • DH （Diffie-Hellman）：一种密钥交换协议，不是加密算法。它允许通信双方在不安全的信道上，通过交换一些公开信息来协商出一个共享的对称密钥，这个密钥随后可用于对称加密算法。

2.2.3 RSA与ECC的比较

🔑 深入理解ECC的优势

表格中的对比具体带来了以下这些优势：

  • 更小的资源占用：由于密钥更短，ECC加密后的数据包也更小。这在网络传输中意味着更低的带宽消耗和更快的传输速度。对于存储空间有限的设备（如智能卡、物联网传感器）来说，ECC也具有天然优势 。

  • 更适合未来趋势：随着移动互联网和物联网的普及，越来越多的设备是资源受限的。ECC能够以更低的计算功耗提供高强度安全，使其成为这些领域的理想选择。测试表明，在Web服务器上使用ECC算法，响应速度可以比RSA快十几倍 。

⚠️ 认识ECC的局限

尽管优势突出，ECC也并非完美。

  • 兼容性考虑：RSA面世更早，因此被几乎所有操作系统和浏览器所支持。而ECC在一些非常老旧的操作系统或设备上（如Windows XP、Android 2.1）可能得不到支持 。

  • 算法复杂性：ECC的数学基础比RSA更为复杂，这给工程师的开发实现和调试带来了一定挑战 。不过，对于使用者来说，这个过程是透明的。

  • 标准化与信任：加密算法的安全性依赖于其具体参数。一些由权威机构（如NIST）标准化的椭圆曲线系列（如美国的“Suit B”算法集）已得到政府和行业的广泛信任和应用 。

💎 如何选择ECC与RSA

了解了它们的区别后，你可以根据以下原则进行选择：

  • 优先选择ECC的情况：当你非常看重性能、效率和带宽时，尤其是在移动应用、物联网项目、需要高性能的服务器或新的数字证书中，ECC通常是更优的选择。

  • RSA依然适用的情况：如果你需要极致的兼容性，以确保你的应用能在任何老旧设备上正常运行，那么RSA因其广泛的兼容性而更稳妥。

3. TLS的过程

以TLS 2.版本为例，共涉及四次握手：

  1. 发送方 发送Client Hello消息（其内容为：生成的第一个随机数 、支持的TLS版本和加密套件套件等内容）

  2. 接收方 发送Sever Hello消息（其内容为：生成的第二个随机数 、支持的TLS版本和加密套件套件 、证书内容（证明自身的合法性）、根据发送方支持的TLS版本和加密套件生成的公钥（Public Key）），其中公钥包含在证书当中。

  3. 发送方根据接收方支持的TLS版本和非对称算法套件利用公钥对生成的第三个随机数加密，此时称加密后的第三个随机数为预主密钥，接着 将预主密钥发送给接收方。

  4. 接收方 利用私钥解析预主密钥，之后 发送方 和 接收方 各自根据先前的第一个随机数 、第二个随机数 和 预主密钥 生成最终的会话密钥，接收方发送“Finished”消息（即发送完成消息），双方随后切换到加密通信过程。

加密通信：之后通信双方根据会话密钥利用对称加密算法进行通信

通过上面的过程可知，TLS是非对称和对称加密算法混用的，第1到4步利用非对称加密算法生成对称密钥 ，第5步再通过该密钥通过对称加密算法进行之后的通信。

上述提及的TLS 1.2涉及两次往返（2-RTT），而TLS 1.3 只需要一次往返（1-RTT）：

3.1 TLS 1.2和1.3的区别

TLS 1.3比起TLS 1.2，少了：通信双方协商、客户端多发送一次客户端DH参数这两个过程。因为TLS 1.3提供的加密方式比起TLS 1.2少了很多，如固定使用DH协商方式时，通信双方一开始就选择好了加密方式，且客户端第一次就可以直接发送DH参数，无需向TLS 1.2一样先进行协商。

3.1.1 TLS 1.2的过程

以使用DH协议为例，TLS 1.2总共需要经历四次握手：

  • 客户端发送Client Hello 消息（包含客户端随机数、密码套件）

  • 服务端发送Server Hello 消息（包含服务端随机数、服务端DH参数、证书和数字签名 ）

  • 客户端发送客户端DH参数

  • 服务端发送 “Finish”消息，双方根据先前的客户端DH参数和服务端DH参数协商生成预主密钥 ，再根据客户端随机数、服务端随机数和预主密钥生成会话密钥随后进入加密通信

3.1.2 TLS 1.3的过程

以使用DH协议为例，TLS 1.3总共需要经历两次握手：

  • 客户端发送Client Hello 消息（包含客户端随机数 和 客户端DH参数）

  • 服务端发送Server Hello、certificate和Server Finished 消息（包含选定的密码套件、服务端随机数、服务端DH参数、证书和”“已完成消息” ），双方随后进入加密通信

3.2 1-RTT和0-RTT

上述过程的TLS 1.3只用了一个RTT便完成了握手过程，这也为0-RTT创造了可能。

0-RTT的过程如下

  • 之前必须先经过了完整的1-RTT握手，生成会话票据（Session Ticket） 或预共享密钥（PSK）

  • 当客户端再次连接时，无需再经过握手（0-RTT），便可以通过PSK导出的“early-data”密钥来对数据进行加密，再加上能够代表上次会话的会话票据（Session Ticket） ，便可以在 Client Hello 中进行发送加密后的数据了（该数据称为“early-data"）；服务端收到数据后也可以使用PSK导出的 “early-data”密钥 对数据进行解密。

  • 注意：early-data”密钥 是用来加密和解密的，而early-data” 指的是用改密钥加密后的数据。
